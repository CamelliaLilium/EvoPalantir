围绕Agent构建一个宏内核的AgentOS。毕竟skills只是作为原子组件，如果skills作为核心，那么是否要赋予skills以运行时状态，skills不再是静态的知识，而是一个运行时概念，具备类似进程上下文的（代码、数据、堆、栈），或者至少要把skills包一层运行时，Agent作为CPU去调度这些进程。skills作为进程的代码/数据段，还需要有运行时数据才能做到“实时分析”吧，比如持久化的skills他就连接一个数据库，并运行一套接口（可以给Agent也可以skills之间编排），或者就要把运行时的职责还给Agent（在进行推演的时候，需要适应性主动中断，就跟OS一样，需要主动中断执行世界对齐线程，此时可人机协同，世界对齐修正），状态属于Agent而不是skills（准确来说是一个world state）。AgentOS，什么是传统OS做不到的（意图计算），什么是不变的（继承OS思想），什么是关键的革新。Agent要获得无限的操作能力，skills作为其上一层的架构就必须要有：进程、内存、文件、IO几个大套。

skills的本体在文件系统，被动态链接到进程中，具备运行时的状态。其实就是OOP了，只不过逻辑不是人类提前写好的，而是 LLM 来实时计算并调用的。

有些肯定不用硬套（堆栈、系统调用之类的），但为了复用老祖宗的智慧，我先来强行映射一波：

1. LLM是CPU，cache专指LLM KV cache，省钱
2. Agent运行时是进程。进程具备PCB，进程可以动态加载各种函数（skills）
3. 每个Agent有独立的内存空间（虚拟内存），即上下文
4. 多 Agent 通信，参考IPC。什么管道、信号量、消息队列、共享内存，几乎所有多智能体通信项目都可以归类到IPC
5. skills，我们把它当成函数（类）。有一个函数/类名（metadata），可以被调用（渐进加载），应当具备独立的状态与方法。静态的数据是共享的，但是必须要有动态状态以增强skills的表达能力，不然它无法替代某些MCP或者带状态的tools
6. 文件系统，得给Agent用，给Agent做适配。那如何优化agent的关键词检索 / 向量检索 / 图检索，总之抽象出来一套文件协议
7. IO，没那么多硬件，skills自己管去吧。不过可以把【Human】抽象成外部设备。可以给你的CPU发中断信号，强行执行“中断处理程序”之类的
