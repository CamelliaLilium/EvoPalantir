## 术语：「消费者」与「消费」

文中 **「消费者」**、**「消费」** 不是本项目新造的业务名词，而是 **软件/数据流语境下的常见说法**：


| 用语          | 含义（本需求中的用法）                                                                                             |
| ----------- | ------------------------------------------------------------------------------------------------------- |
| **消费者**     | 在 **主数据流或产品链路中**，**必须接收**「校正记忆 / 经验库」产出物并加以利用的 **下游环节或角色**。与消息队列、管道、分层架构里 **Consumer / downstream** 同义。 |
| **消费（产出物）** | 将上述产出物 **作为输入使用**（读入、展示、参与计算），**不是**经济学里的「消费支出」，也 **不隐含**「用后即弃」或「销毁数据」。                                 |


若更易读，可与下列说法 **互换理解**：**必选下游**、**须接入的环节**、**必须使用这些输出的系统/界面**。

## 2. 背景与目标

- 针对现实对齐方案中的 `**calibration_memory_engine`（校正记忆）** 及类似 **经验库** 能力：记录预测与现实的差距、校正规则与效果、支撑相似案例与后续预测/推演。
- 本轮优先 **丰富调研、对齐场景**，形成 **可落地的设计框架**（调研偏重）。

## 4. 功能需求（V1 口径，与知识库一致）

### 4.1 必选下游（原「消费者」）

- **推演侧：** `forecast_runtime`（或等价预测/推演链路）须能 **接收并使用** 校正记忆产出的经验/相似案例参考（作为输入参与后续计算或提示）。
- **人读侧：** 须有 **报告类输出**（含实验对照、误差解释等）；**呈现形态**（面板/导出/静态页等）**不在本需求中裁定**，在《调研与设计文档》中确定。

### 4.2 相似案例

在《需求说明》中仅约定 **业务目标**：在同类事件/阶段等情境下能 **召回可对齐的历史校正记录** 供对照；**不绑定**具体相似度算法、嵌入或实现细节。

### 4.3 与知识库一致的行为约束

不自动改写主业务逻辑；先 **记录、检索、总结**（以现实对齐方案 §4.10 为准）。

## 6. 范围边界（本轮交付）

### 6.1 本轮 **明确包含**

- 本《需求说明》定稿。
- 《调研与设计文档》（调研扩展 + 符合场景的设计框架，并引用/对齐 `knowledge/` 与 `rag_design` 锚点文档）。

### 6.2 本轮 **交付形态**

- **优先交付** 上述 **文字与图表设计**；**不以「预先排除某种实现深度」为代价** 做次优设计。
- **合并进主仓库的实现代码** 是否在本轮或紧随其后的迭代中完成，由 **《调研与设计文档》与后续实现计划** 在 **设计最优、资源与依赖可行** 的前提下 **另行约定**，本需求说明 **不预先排除** 含 **V2 记忆栈**（如 MemOS、Mem0 等）在内的 **深度集成实现**，亦 **不强制** 本轮必须交付代码。

### 6.3 与外部系统的关系

- 对 MemOS、OpenViking、Mem0 等 **具体 API、能力** 须 **查官方文档/源码核实**，禁止无来源断言（见 `rag-design-range.mdc`）。
